CDT 1.2.1 status:

Over 50 defects have been made to the 1.2 stream since it went GA back in October 2003. The general concensus was that January 2004 was an appropriate time to make 1.2.1 generally available. In the meantime, a 1.2.1 milestone build will be made available before the end of the year so that users can pick up the many defects that have been fixed so far. We will firm up a GA date at the January CDT conference call scheduled for the first Wednesday in January (January 7, 2004).

 

CDT 2.0 Status:

  • The proposed Content Assist Enhancements feature has been to the committed section of the 2.0 plan as IBM-Rational has committed the resources for the associated development, testing and documentation.

     

  • QNX has similarly committed to the Class Browser feature, and that feature should be moved to the committed section as well.

     

  • There were a couple of general questions about what it meant for a CDT contributor to "commit" to a feature and thus have it moved from the proposed section of the plan to the committed section. The answer is that committing means committing to develop, test and document the feature. A note will be posted to the cdt-dev mailing list re-iterating this so that is clear for everyone.

     

  • During the recent CDT 2.0 kick-off in November, a multitude of potential usability enhancements were discussed and proposed. A proposed plan item was subsequently created in bugzilla that captured the shopping list. IBM-Rational will be taking on a small number of these enhancements, and will be creating a scaled down bugzilla plan item to put in the committed section of the 2.0 plan. There are a bunch of really good ideas still left in that shopping list that provide a great opportunity for other contributors to pick up and get involved!

     

  • CDT M5 has been achieved a few days after the Eclipse project reached M5 on the Eclipse 3.0 release.

 

Documentation

The re-structuring of the documentation (on-line help) has been completed so that all contributors can contribute to the documentation by submitting patches (same process as for submitting code changes). An announcement of this will soon be posted to the appropriate mailing lists along with a style guide that can help documentation contributors maintain consistency across the docs.

 

Verification/Bugs:

Not much to discuss here since we are in the early stages of the 2.0 release.

 

Infrastructure/Builds:

The automation of the regular builds is broken :-(

One build (the M5 build) has been already done, and another will be manually done in December (likely on the M6 build from Eclipse). Doug Schaefer is looking into what's causing the trouble and how to get the automated builds back up and running again.
