July 2004 Project Meeting Minutes
CDT Meeting Minutes

July7, 2004

Phone in meeting of Eclipse CDT stakeholders. Review progress to date and map out the future direction for the Eclipse CDT Project.

1:00 PM EST on July 7, 2004

 View the Agenda

 

Minutes
 

Here are the minutes of the CDT conference call.
 

Present:

QNX, IBM-Rational, RedHat, Intel, MontaVista, Texas Instruments, Rockwell, WindRiver

 

Update from participants:

QNX:

  1. Starting to work on debugger enhancements to catch up to Eclipse 3.0 that did not make it into CDT 2.0, and also on some managed build system generalizations. This is feature work and will be delivered in the HEAD stream for release 2.1. This work WILL cause API churn.
     

IBM-Rational:

  1. Working on performance improvements and other defects that are required for a 2.0 maintenance release. Next CDT based product release (fall 2004) will be based on the 2.0 maintenance release.
     
  2. Will continue with testing of 2.0 and 2.0.1 stream, especially in areas not rigorously tested by us to date.
     

Rockwell:

  1. Still working on getting corporate approval  to contribute to open source and the CDT project.
     
  2. Doing some plug-in development that they would like to contribute (pending approval) in the areas of:
        - C++ code formatter
        - PC Link Integration
     
  3. Availability of ClearCase plugin for Eclipse 3.0 (due Sept. 2004) gating internal adoption of Eclipse 3.0 and CDT 2.0
     

Texas Instruments:

  1. Concentrating on product due to ship this fall, based on CDT 2.0. Will be keeping an eye on the 2.0 maintenance stream and 2.0.1 release to gauge wether or not to pick some fixed up or stick with 2.0.
     

Redhat:

  1. Awating resolutin of how & where to contribute RPM and Profiling plugins that have been developed.
     
  2. Working CDT 2.0 into next product release schedule for August 2004
     

Intel:

  1. Have proposed some managed build enhancements and are looking to work with community to implement them.
     
  2. Have raised some error parser issues that should be resolved, not necessarily in the 2.0 maintenance stream but definitely for a subsequent release.
     

MontaVista:

  1. Moving forward to CDT 2.0 internally
     

Wind River:

  1. Looking at CDT 2.0 and comparing capability with their previous IDE (Eclipse 2.x based).
     
  2. Interested in additional functionality & looking to collaborate on Refactoring and Managed Build enhancements

 

CDT 2.0 Maintenance Stream Plan & Process :

  • General agreement that the 2.0 branch should stay frozen from an API perspective and remain backwards compatible.
  • Defect fixing only, and all commits to the branch should reference the defects being addressed by the changes.
  • All defects that will be fixed in the 2.0 branch should have the Bugzilla target_milestone field set to 2.0.1
  • Any potential API changes/breaks should be communicated and discussed within the community PRIOR to committing them.

 

CDT 2.1 Content & Timing

  • To help the CDT community plan for on-going releases, it was proposed that the community should put in place a formal planning exercise on a six month cycle.
  • Current view is that a 2.1 release would be required by the end of October, primarily in support of a QNX product release.
  • Outside of the formal 6-month planning session, plans should be captured and discussed using Bugzilla plan items. Each of these plan items should capture what is being proposed and when it is expected to be delivered.
     

Contributing RPM and Profiling Plugins (Red Hat)

  • 2 possibilities were proposed: a) create a separate project or b) create a new CDT component with the Red Hat contributors set up as the initial committers. It was felt that creating a separate project for this is too heavyweight, so it was agreed to proceed with option b.
  • Sebastien Marineau will formally request the creation of the new component.
  • As with the CppUnit, these plugins will be hosted on the CDT community page, not on the main download page.

 

Managed Build system proposal

  • Many CDT participants are interested in generalizing the managed build system so that we can all share it as much as possible and avoid each company doing their own. Several proposals have been made along the way. Sean Evoy, who has been quarterbacking most of the work done to date on MB will set up a meeting within the next couple of weeks with the goal of identifying and assigning specific pieces of work to help achieve this goal. Stay tuned to the cdt-dev@eclipse.org mailing list for more information on this.

 

Error Parser issues

  • Intel has proposed to fix several error parser issues (Bugzilla items: 65576, 66762, 66808). Alain Magloire at QNX has some additional suggestions and will work with Intel on getting this stuff implemented. It is not required for the 2.0 maintenance branch, although there is one piece of low hanging fruit that could be addressed on the maintenance branch (restoring defaults).

 

Any comments, additions, corrections etc. welcome! As always, many thanks to everyone for your ongoing support.
