  1. What is the CDT?

    The CDT (C/C++ Development Tools) Project is working towards providing a fully functional C and C++ Integrated Development Environment (IDE). Our focus is development on Linux for deployment on Linux, but we are interested in participation from others who would like to extend our work in other directions. The CDT is fully open-source and implemented purely in java as a set of plugins to the Eclipse platform. To learn more visit the CDT Home Page.

  2. How is the CDT licensed?

    The CDT consists of software produced by the CDT team combined with third party software developed from other Open Source Projects. The software produced by the CDT team is licensed under the Common Public License. The software designed by third parties is made available under their respective licenses. Refer to the about.html file in the root directory of every CDT plugin for specific licensing information.

  3. How is the CDT Project organized?

    The CDT was the first approved subproject of the Eclipse Tools Project hosted at www.eclipse.org/tools. To learn more about the organization of the Eclipse Tools Project, read the Eclipse Tools Charter.

    The CDT project itself is organized into autonomous components each with its own set of committers, bug categories and mailing lists.

  4. How do I find out about future releases of the CDT?

    Release Milestones (which represent planned availability dates for Stable CDT builds) are noted in the CDT Plan Documents which can be found on the CDT Overview Page. To see the currently available CDT builds, visit the CDT Download Page.

  5. How do I ask questions?

    CDT related questions that are not answered in this FAQ or the documentation should be posted to the CDT newsgroup. You will need a password. You can also use this simple web interface to browse the newsgroup.

    General Questions about the Eclipse SDK which includes the Eclipse Platform , JDT (Java Development Tools), or PDE (Plugin Development Environment) should be posted to the Eclipse newsgroup.

    Keep in mind that these newsgroups are public, so do not include any confidental information in your questions. You should also read How to ask questions the smart way by Eric Raymond before participating in the newsgroups. NOTE: Please submit bugs to bugzilla, not to the newsgroups. See the How do I report a bug or request a feature? section of this document.

  6. How do I report a bug or request a feature?

    The CDT Project (like the Eclipse Project) uses bugzilla as its bug and feature tracking system. Entering a bug\feature report is as simple as filling in a web form on the eclipse bugzilla page. The first time you enter a bug you will need to create a new bugzilla account for yourself by providing an email address and choosing a password.

    Before enterring a bug report, you should search bugzilla to see if someone else has already entered a bug report similar to yours. If you find a bug report that outlines the problem you are seeing, you can simply annotate it with your comments to let the developers know that you have also hit the bug. Also you can add yourself to the CC list of the bug so that you will notified when the status of the bug changes or someone adds comments.

    Once you have searched bugzilla and not found anything, you can go ahead and enter a new bug report. Please read the bug writing guidelines located on the eclipse bug reporting page.

    To make your bug report more helpful include the following in your bug reports:

    1. The build level of both the CDT and Eclipse that you are using. For example, "cdt 20020508 with Eclipse M5 Stable Build"
    2. Your computer's specifications (OS + level, memory, other pertinent info)
    3. The contents of your .log file (or lack thereof). This is especially important if you get a dialog that reports an internal error. See What is this .log file I hear so much about? for information on on finding your .log file.
    4. The Java runtime or development kit you are using to run eclipse (use java -version or java -fullversion)
    5. A description of what you were doing when you encountered the bug
    6. An explanation of how the observed behavior differs from the expected behavior

  7. What is this .log file I hear so much about?

    The .log file is used by the Eclipse Platform to log runtime errors. It is invaluable when debugging both problems with plugins not starting, as well as any exceptions they may throw once loaded. A .log file should always be included when opening a bug against the CDT or Eclipse (or if there is no .log file, that fact should be mentioned in the bug).

    The .log file is located in the workspace/.metadata directory (or sometimes just in the .metadata directory).

  1. Which CDT build should I download?

    There are currently 2 versions of the CDT:
      R1 - for use with the Eclipse Release R1.0.
      R2 - for which we provide periodic builds that are for use with specific Eclipse Stable R2.0 Builds.

    So,
    • If you are using Eclipse Release R1.0, the you should download CDT R1.
    • If you are using a Stable R2 Eclipse Build, then check the CDT Download Page for information on which CDT Build to use.
    • If you are using an Integration or Nightly Eclipse Build, then you can download the closest CDT R2 match, but you do so at your own risk since it is possible that Eclipse has changed enough since the last Stable build to break the CDT.

  3. Why isn't the XXX operating system supported by CDT? (XXX usually == Windows, but may take on other values)

    "Supported" has a particular meaning to us. It means that on that platform we have a good level of confidence that CDT works correctly and that its function is appropriate and complete. That means

    • someone has ensured that the function addesses the important use cases
    • the function is exercised by regular execution of a test plan
    • identified problems are tracked to resolution
    • there is a recipient for user feedback
    • code patches are developed as necessary to correct or extend CDT on that platform
    To make this all happen a platform has an Owner - someone who accepts the responsibility to make sure those things all happen.

    The IBM CDT team owns Linux support but doesn't have the capacity to own other platforms. That doesn't mean we don't care about them. It certainly doesn't mean we won't accept code changes for them - it is already the case that much of CDT will work outside of Linux and there is non-Linux code in CDT to accomplish that.
    It does mean that we can't "own" those platforms, and without an owner we don't have the necessary confidence to claim support.

    If this answer makes you want to volunteer to own a platform, that is the reaction we were hoping for!

  4. I can't find the CDT download for Windows, isn't there one?

    There is no special download for Windows. Much of the CDT function will work on Windows, but in some cases we are leveraging Linux system tools for things like debugging (gdb), autoconf support, and profiling (gprof). So this function will not work "out of the box" on Windows.

    However, installing the Cygwin toolkit, which is a Unix environment for Windows, gets you most of this function.

  7. How do I uninstall CDT?

    We do not currently ship an uninstaller. Once we have moved up to the latest "Update" support in Eclipse this should be taken care of. In the meantime you can uninstall the CDT in 2 manual steps:

    1. Remove the CDT plugins
      • Navigate to the directory where the CDT plugins are installed (typically eclipse/plugins)
      • Remove the following plugins:
        • org.eclipse.cdt.*
        • com.ibm.debug*
        • com.ibm.lpex*
    2. Remove the CDT metadata
      • Navigate to the workspace/.metadata/.plugins directory
      • Remove the following directories (if they exist):
        • org.eclipse.cdt.*
        • com.ibm.debug*
        • com.ibm.lpex*
