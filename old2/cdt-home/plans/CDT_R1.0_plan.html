C/C++ Development Tools - CDT Project
R1.0 Plan

Last revised February 27, 2002
First edition January 23, 2002
This document lays out the plan for the next major release of the C/C++ Development Tools (CDT) Project, designated release 1.0. The main emphasis of the 1.0 release will be to deliver features judged to be required for a 1.0 release and on improving quality (bug fixing and more complete test coverage).  Note that the CDT 1.0 release will only run on Eclipse R1.0.

This plan serves as an important input to the planning processes of the various customers of the CDT Project. It tells them what to expect (or what not to expect) in the next release so that they can plan their projects accordingly.

Plans are rarely completely unchanging over the life cycle of a release, and this project is no exception. We will update this document as required to reflect any changes to the 1.0 release plan. We reserve the right to add items to this list. The developer mailing list archives contain further, in-depth information about how a component is changing. Particular problem reports being worked on for the 1.0 release can be tracked through the Eclipse bug database; the "Target Milestone" field reflects when a work item will become available in the main development stream.

Release deliverables

The release deliverables are:
  • Source code release for CDT project, available as versions tagged "R1_0" in the Eclipse CVS repository.
  • CDT Project drivers (downloadable).

Compatibility with Previous Release


API Contract Compatibility: CDT 1.0 will be upwards contract compatible with CDT 0.5 unless noted. This means that programs in full compliance with contracts specified in the CDT 0.5 APIs will automatically be in full compliance with the CDT 1.0 APIs. Downward contract compatibility is not supported. There is no guarantee that compliance with the CDT 1.0 APIs would ensure compliance with the CDT 0.5 APIs.

Work Items

Note that fixing bugs, improving test coverage, documentation, examples, etc. are considered routine ongoing maintenance activities and are not included in this plan unless they would also involve significant changes to the plugin.

The following committed items reflect new features of CDT, or areas where existing features will be significantly reworked.

    General

Standalone Installer.  A simple standalone graphical installer will be provided that removes any existing CDT installations, and installs the new version.  It will also provide some options, such as unzipping the source for each plugin, building each plugin, and perhaps using ftp to put the server pieces on a remote machine and run them.

Support for appending environment variables.  Support will be added to allow a Project's Environment to properly override any system environment variables using variable references (such as $PATH, or ${FOO}, or %BAR%)

org.eclipse rename.  All plugins shipped with CDT except for LPEX and the debugger will have their package prefix names changed from com.ibm to org.eclipse.

Network failure diagnose and report.  In the event of a network failure, when users are working with remote projects, the connection is gracefully terminated and a message reports the occurrence.

Customizable command specification in menu.  In the C/C++ Projects view, context sensitive menus display a submenu for "Command Specifications".  The submenu is populated with menu actions for launching each of the command specifications that were defined in the "Command Specifications" view.

Browse toolbar actions for declarations views.  The Project Objects, System Objects, and Parse Files views have toolbar items with actions that facilitate view browsing with history by changing the input to the corresponding view.   The "home" toolbar item navigates to the default view input, the "back" toolbar item browses back in the history, the "forward" toolbar item browses forward in the history and the "step into" toolbar item browses into the selected object.

Standard Input Support In Output View  Allow users to interact with programs in the Output View.

New icon set.  A standard set of plugin icons more consistent with existing Eclipse icons will replace those that are currently used.  The label provider that accessed icons based on their name changes to use defined mappings between names and icons.

Byte transfer from client to server. A new set of DataStore APIs is added to facilitate transfer of client files to a server.  These APIs allow the transfer of variable sized sets of bytes to the remote side and appending to existing remote files.
 

    Autoconf
Enable/Disable advanced autoconf actions.  Based on the Target type in each Makefile.am and the location of the Makefile.am some of the actions  will be disabled/enabled.  As an example: The "Create Program Makefile.am" will be disabled if the Makefile.am in this directory is in fact of type Programs Makefile.am.  Another example is the action Create TopLevel Makefile.am will be enabled only for the top level directory.

On/Off automatic autoconf .am update.  A feature that lets the user decide whether or not to let the system automatically update dependencies between Makefile.am's.   The other alternative is the user manually maintains all Makefile.am's.
 

    C/C++ Parser
Parser declaration/definition distinction.  In the 0.5.0 release, if a function has a definition separate from its declaration, then there is a function object created for each.  This redundancy will be addressed.

C/C++ Preprocessor upgrades.  The parser will be upgraded to provide macro uses and conditional inclusion.
 

    Debug
Working directory on run/debug launch.  Instead of starting a selected executable from the project directory by default, the user now has the capability to specify a different directory as the working directory.
 
Debugger detach.  The user has the capability to start a debug session by attaching the session to a running process.  Upon termination of this debug session, the debug engine will simply detach from the process.  It will not terminate the process.

Debugger Disassembly view. Disassembly views are now generated dynamically as user steps into different sections of the codes.  The engine will automatically provide the user with the disassembly view if a source file is not successfully located
 

     Help Infrastructure
Access help from parser views. Access to Find Help from functions/classes objects.

Filtering preferences for help. Block entries that match a regular expression pattern from being returned by a search request.

Topic indexing for help. Present an appropriate description for each html result returned by a search.
 

    Documentation
Javadoc for extension APIs. Provide javadoc comments and tags in public extension APIs for DataStore tooling framework.

Tutorial - Autoconf. A walk through showing how to manage C/C++ projects from scratch. Also, another walk through of importing an existing managed project and how to run it using the autoconf features.
