This document lays out the feature and API set for the next feature release designated release 2.0.

Plans do not materialize out of nowhere, nor are they entirely static. To ensure the planning process is transparent and open to the entire Eclipse community, we (the Eclipse PMC) post plans in an embryonic form and revise them throughout the release cycle.

The first part of the plan deals with the important matters of release deliverables, release milestones, target operating environments, and release-to-release compatibility. These are all things that need to be clear for any release, even if no features were to change. 

The remainder of the plan consists of plan items for the various Eclipse subprojects. Each plan item covers a feature or API that is to be added to Eclipse, or some aspect of Eclipse that is to be improved. Each plan item has its own entry in the Eclipse bugzilla database, with a title and a concise summary (usually a single paragraph) that explains the work item at a suitably high enough level so that everyone can readily understand what the work item is without having to understand the nitty-gritty detail.

Not all plan items represent the same amount of work; some may be quite large, others, quite small. Some plan items may involve work that is localized to a single Platform component; others may involve coordinated changes to several components; other may pervade the entire Platform. Although some plan items are for work that is more pressing that others, the plan items appear in no particular order.

With the previous release as the starting point, this is the plan for how we will enhance and improve it. Fixing bugs, improving test coverage, documentation, examples, performance, usability, etc. are considered routine ongoing maintenance activities and are not included in this plan unless they would also involve a significant change to the API or feature set, or involve a significant amount of work. All interesting feature work is accounted for in this plan.

The current status of each plan item is noted:

  • Committed plan item - A committed plan item is one that we have decided to address for the release.
  • Proposed plan item - A proposed plan item is one that we are considering addressing for the release. Although we are actively investigating it, we are not yet in a position to commit to it, or to say that we won't be able to address it. After due consideration, a proposal will either be committed, deferred, or rejected.
  • Deferred plan item - A reasonable proposal that will not make it in to this release for some reason is marked as deferred with a brief note as to why it was deferred. Deferred plan items may resurface as committed plan items at a later point, especially if someone volunteers to carry out the item.
  • Rejected plan item - Plan items that were proposed but judged unworkable are marked as rejected plan items, with an accompanying summary of why they were dismissed. Keeping track of rejected items avoids repeating the discussion.

Release Deliverables

The release deliverables include the following:

  • Source code release for CDT available as versions tagged "CDT_R2_0" in the Eclipse Tools CVS Repository
  • Various CDT features released on the Eclipse Update Site as well as distribution zips.

Release Milestones

Release milestones occuring at roughly 6 week intervals exist to facilitate course-grained planning and staging.  These milestones are intended to line up with equivalent milestones in the Eclipse 3.0 Project Plan.

  • Friday, November 21, 2003 - Milestone 1 (2.0 M1) - stable build reflecting progress. Runs on Eclipse 3.0 M5.
  • Friday, January 16, 2004 - Milestone 2 (2.0 M2) - stable build reflecting progress. Runs on Eclipse 3.0 M6.
  • Friday, February 27, 2004 - Milestone 3/Release Candidate 0 (2.0 RC0) - freeze date, only high priority defects allowed going forward.
  • Friday, April 23, 2004 - General Availability (2.0 GA) - release deliverables available for download (8 weeks after RC0)

Currently, the Eclipse 3.0 plan only sets out M6. This plan assumes the next milestone will be RC0. If this is not the case, the CDT plan will be changed to line up with the Eclipse 3.0 plan.

Internationalization

The CDT is designed as the basis for internationalized products. The user interface elements provided by the CDT, including dialogs and error messages, are externalized. The English strings are provided as the default resource bundles.

Latin-1 locales are supported by the CDT on all of the above operating environments; DBCS locales are supported by the Eclipse SDK on the Windows, GTK, and Motif window systems; BIDI locales are supported by the Eclipse SDK only on Windows operating environments.

Compatibility with Previous Releases

One of the main themes for this release of the CDT will be solidifying and documenting the APIs to various functions in the CDT. As such, no committment is made to maintain compatibility with previous releases. Every effort will be made to maximize compatibility, and area that do require change will be documented and communicated to the community.

Plan Items

The plan items listed in this section reflect new features of the CDT, or areas where existing features will be significantly reworked. Numbers in parentheses link to bugzilla problem reports for that plan item (query bugzilla for all CDT 2.0 plan items).

**Note: The eclipse project follows the convention of creating a bugzilla entry (query bugzilla for all Eclipse 3.0 plan items) for all planning items, which is essentially a defect report with the following characteristics:
        -  severity = enhancement
        -  keyword = plan
        -  summary begins with "[plan item]
        -  target = Release in which the plan item will be delivered

The items are grouped by theme to reflect the two key customers of the CDT - end users and ISVs wishing to integrate their tools with the CDT. The themes are: 

  • User Experience Theme - improve the experience of the user writing C/C++ code.
  • ISV Enablement Theme - improve the ability for ISVs and other developers to create tools that interact with the CDT and each other.

Theme: User Experience

Committed Items

Eclipse 3.0 support. Aligning with Eclipse 3.0 is the main task for this release. This means more than just getting the CDT to compile and run on Eclipse 3.0. It also means ensuring the CDT provides the same look at feel and supports a number of the new features that are being provided by the Eclipse Platform.

Rename refactoring. This feature represents our first foray into the world of refactoring. This includes bringing in a refactoring framework that is very similar to the JDT's refactoring framework. The focus is then on rename refactoring which will leverage off and make improvements to the C/C++ Search feature to correctly refactor the references.

Content assist enhancements. The code completion part of the contents assist feature will be enhanced for performance, handling of C visibility rules, and for the handling of C++ code completion on members.  The majority of this work will be to make more use of the parser to provide accurate and speedy competion proposals.

Managed Build enhancements. The managed build feature will be enhanced for usability and for flexibility to support a large variety of build target environments as well as to better support multiple configurations of tool and tool settings. Support will also be added for handling different variety of underlying tools including different make utilities, as well as support for different file types (like lex/yacc).

User Documentation enhancements. The user documentation will be expanded on to ensure the success of new and experienced users of the CDT.

Internationalization and Accessibility. Normally, these capabilities should be designed into features as they have been developed. Areas where this hasn't be done will be addressed in this release.

Proposed Items

C/C++ Search enhancements. This feature will be enhanced to add the capability of selecting a name in the editor and searching for declarations, definitions, and references to that name. The parser will be used to determine accurately the appropriate C/C++ definition of the name to be used in the search.

C++ Class Browser/C++ Hierarchy View.  A view dedicated to the navigation of C++ class hierarchies associated with objects or classes defined in a project. This include the ability to see class members, their specifications (public, private, protected, virtual, etc), their overrides, and the inheritance structure (both inherited from and inherited by).

Deferred Items

Quick Fix.  Do something like the Quick Fix feature in Java.

Mixed Java/C/C++ Development.  Support the writing, building, and debugging of native methods in a mixed Java/C/C++ environment. If we want to get real good, support the debugging of an embedded JVM in a C/C++ application. Java calling C, C calling Java...

Rejected Items

None at this time.

Theme: ISV Enablement

Committed Items

ISV documentation. A new help plugin will be created that will provide documentation that will help ISVs who wish to integrate with different aspects of the CDT. This document will provide a reference section

Proposed Items

Parser extensibility. To support different language variants as supported by different compilers, a mechanism needs to be in place to change the parser to adapt to those variants. This will ensure accurate results for Search, Content Assist, and other features that rely on the parser.

Documentation extensibility. To support providing the user documentation during content assist and text hover, an extensibility mechanism will need to be put in place to support different documentation content providers.

Tools integration extensibility. To support integration with different tools for different tasks, a model describing these tools (sometimes called a Target model) is required. This model would provide such information as command name and available options. It would also provide information for specific parts of the CDT such as providing a mechanism to provide default symbols and inclusion paths for a compiler.
