This document lays out the plan for the next major release of the C/C++ Development Tools (CDT) Project, designated release 1.0. The two main goals of the 1.0 release will be to enable CDT to run on Eclipse R2.0.1, and to define and implement a fully extensible framework.
Note: The CDT 1.0 release will only run on Eclipse R2.0.1.

This plan serves as an important input to the planning processes of the various customers of the CDT Project. It tells them what to expect (or what not to expect) in the next release so that they can plan their projects accordingly.

Plans are rarely completely unchanging over the life cycle of a release, and this project is no exception. We will update this document as required to reflect any changes to the plan. We reserve the right to add items to this list. The developer mailing list archives contain further, in-depth information about how a component is changing. Particular problem reports being worked on for this release can be tracked through the Eclipse bug database; the "Target Date" field reflects when a work item will become available in the main development stream.

Note:Version 1.0 APIs are NOT frozen. They are expected to evolve over the next 6-9 months. Every effort is made to allow for backward compatibility, but backward compatibily cannot be guaranteed.

CDT Debugger


The draft plan for the debugger/launcher components. The plan is split into 3 major "releases". The first is in october, as the first, working prototype implementation, the second is targetted for march 2003, and the third for august 2003.
The focus is on creating a first working prototype of the CDT debugger, that works with a gdb/mi backend. The debugger includes standard functionality such as source file integration, stepping, variables, etc. as well as C-specific views such as registers, assembly and memory. It also includes an initial version of the C Debug Interface (CDI) that is used to programmatically control the CDT debugger.

Compatibility with Previous Release


API Contract Compatibility: The architecture for the CDT 1.0 is completely different than the architecture of the previous (prior to July 2002) CDT 1.0 release. As a result, there will be no compatibility supported, between previous version 1.0 and the new version 1.0.

Future releases will strive to be contract compatibile built upon the new 1.0 contracts. Given that the 1.0 API contracts are not frozen, full contract compatibility is not guaranteed between 1.0 and 2.0 releases. This means that programs in full compliance with contracts specified in the CDT 1.0 APIs as frozen, will automatically be in full compliance with the CDT 2.0 APIs. Contracts for API that are not listed as frozen will not be guaranteed.
Downward contract compatibility is not supported. There is no guarantee that compliance with the CDT 2.0 APIs would ensure compliance with the CDT 1.0 APIs.
