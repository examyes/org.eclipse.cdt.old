C/C++ Development Tools 1.2 Plan

 Sept 10, 2003

This document lays out the plan for the next major release of the C/C++ Development Tools (CDT) Project, designated release 1.2. The two main goals of the 1.2 release will be to round out the editing, searching and indexing capabilities of the CDT, and to round out the CDT debugger.
Note: The CDT 1.2 release will only run on Eclipse R2.1.

This plan serves as an important input to the planning processes of the various customers of the CDT Project. It tells them what to expect (or what not to expect) in the next release so that they can plan their projects accordingly.

Plans are rarely completely unchanging over the life cycle of a release, and this project is no exception. We will update this document as required to reflect any changes to the plan. We reserve the right to add items to this list. The developer mailing list archives contain further, in-depth information about how a component is changing. Particular problem reports being worked on for this release can be tracked through the Eclipse bug database; the "Target Date" field reflects when a work item will become available in the main development stream.


 

Editing Capabilities
 

  • Search for references
    • Allows the user to search for different kinds of references to a specific variable/function/object/method. Types of matches to look for could include (Declarations, Implementers, References, All Occurrences, etc.)
  • New Class Creation Wizard
    • A wizard to automate the creating of a new class
  • Navigation Assistance
    • C/C++ Outline View
      • Filters to show/hide things like static members, non-public members, fields, etc.
      • Open symbol’s source (open the header/cpp file where this source is “defined”)
      • Open header file referenced by a #include (may require build system support to determine inclusion paths)
    • C/C++ Projects view
    • General cleanup to bring in line with Eclipse 2.1 Navigator functionality


Build Capabilities

  • Continue progress on Managed Make Builder
    • Build a project using automatically generated makefile(s)


Debug Capabilities

  • Breakpoints
    • Defered Breakpoint(If the underlying debugger supports shared library loading notification).
    • Investigate action to set breakpoint for specific thread(if the underlying debugger supports it) and session.
    • Setting breakpoints on external sources with persistency on the project.
  • Presentation
    • Formating values (hex, octal, ...)
    • Detail panel for Variable view.
    • Casting variables to different types in the Variable view.
    • Investigate a way to make static variables available even when not in the context of the stackframe.
  • Debug protocol
    • Support for GDB/MI 2 (still support GDB/MI 1). This will allow using the new MI2 features which will be found in gdb 6.0.
    • Investigate variables book-keeping.  The rationale; on some embedded boards probing the variables at every step can be to intrusive and sometimes destructive for example, accessing a volatile.
  • CDI
    • Documentation for the C/C++ Debug Interface(CDI)
    • Commit the cdi-spy plugin as an example, the cdi-spy resembles jdt-spy, it listens to all debug events(breakpoint, dll loading etc ..) and presents the informaiton in a nice Treeview.


Documentation

  • Create a user's guide for the CDT as it stands today (release 1.1) and begin documenting the capabilities being added in 1.2 as time permits. A preview of the Table of Contents for this user's guide can be found here.
  • Update the user FAQs and HOWTOs on the web site.


Testing

        The IBM/Rational team will be dedicating some resources to test all of their contributions to version 1.2 of  the CDT. In addition to that, we will continue broadening of CDT testing to address areas that have not yet received any real test coverage. Finally, an area of focus will also include CDT scalability with large projects.
 

Work Items

Note that fixing bugs, improving test coverage, documentation, examples, etc. are considered routine ongoing maintenance activities and are not included in this plan unless they would also involve significant changes to the plugin.
 

Timeframe

The timeframe for the CDT 1.2 release is mid-september. The feature complete, freeze and GA target dates are
as follows:
 

  • Milestone 1 (M1) :                  September 19, 2003
  • Freeze 1 (F1):                       October 1, 2003
  • Release Candidate:                October 8, 2003
  • General Availability (GA):        October 15, 2003
The CDT 2.0 release will be the next major release following the CDT 1.2 release. It is currently planned for the 2nd Quarter of 2004, in sync with Eclipse 3.0.
Major features targeted for CDT 2.0 are outlined below.
 

Compatibility with Previous Release

API Contract Compatibility: The architecture for the CDT 1.2 is a little different than the architecture of the previous releases. As a result, there will be some
API incompatibilitie with previous versions. As we work through integrations with a wider variety of tools and round out the CDT functionality, these APIs will freeze; this is one of the CDT 2.0 objectives.

 

CDT 2.0 release

The CDT 2.0 release is planned for Q2/2004, and will focus on the following areas:

  • Refactoring - the ability to make system-wide code changes, optionally previewing the changes that will result from the refactoring operation.
  • Writable DOM - this will be required in order to support user-level capabilities like refactoring.
  • Managed Make System and build model - further work to firm up the build model APIs, as well as complete the native CDT builder.
  • Investigate and possibly develop a non-MI debugger back-end, for example a native WIN32 debugger implementation.
  • Investigate integration of the CDT and JDT debug capabilities, t provide seamless debugging of mixed C/Java apps.
  • Freeze the CDT integration APIs, ensuring compatibility between future versions of CDT and other integrated tools.
