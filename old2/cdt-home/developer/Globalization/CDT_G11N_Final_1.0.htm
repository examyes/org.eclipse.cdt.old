Summary

The results from 3 of the 4 stages of internationalization: String Externalization, Pseudo Translation, Enablement Testing can be found from the previous report, CDT 2.0 Globalization Interim Report, while the final stage of internationalization (Cultural Formatting), and the Localization results are found in this report. 

Internationalization Phase 4: Cultural Formatting

The last phase of internationalization testing was completed at the end of June: verifying the correct cultural formatting of the display text. For instance, a number with commas and decimals in the US has the commas and decimals reversed in France. Date and time preferences differ per country as well. Sorting has different levels of priorities whether or not accents and capitalization is involved. For instance, ‘a’ with umlaut could be an ‘a’ with an accent in one country, so it would sort with the ‘a’s, while in another country it would be a completely different letter found after ‘z’.

Localization 

Localization includes the translation of the externalized strings and Translation Verification Testing (TVT). Note that localization is not only text translation but some colors or images may be undesired by some countries and would be 'translated' as well. The translations have been packaged by into a single zip which can be used to start eclipse in any language. Each plugin has an NL fragment with the same name as the associated plugin + "_nls1" before the version number. This fragment contains the translations for this plugin for at least one language. The collection of all the fragments for a particular language is called a localization pack. CDT won’t need to be recompiled, just unzip the fragments and start eclipse.

TVT was performed in the 9 countries of translation during the whole month of July on Eclipse and Eclipse project tools including CDT. TVT ensures that there are no English strings, the GUI handles the expanded strings, the translations make sense, the F1 help shortcuts work, and mnemonics and other shortcuts work.

Globalization

If a product is internationalized, and then localized for a particular country, then globalization is not fully achieved until that country can access any localization pack the product supports. If I'm in Germany, I may still want to run Eclipse in French or Japanese. The zip files posted on the eclipse download site each contain the nine localization packs for all the program data (text displayed). The supported operating systems themselves contain additional localization packs for culturing sorting as well as date and number formats. See the section, Eclipse Startup, for how to invoke these localization packs.

Eclipse NL Startup

There are several localization packs to be picked up by the application in order to see and do everything you want in your chosen language. The NL packs provided by eclipse and CDT are the program data localization packs which contain the text translations. The localization packs for the formats of date, time, numbers, collation, OS messages such as browse dialogs are provided by the OS.

Invoking OS Localization Pack

The OS localization packs Eclipse depends on are Date, Time, Number, Messages(Dialogs), Collation, Character Type.

Windows: In the Control panel: Regional Options: set the locale (on W2K) or Standards and formats (on XP) to the desired language for dates/number formatting. Set the menus and dialogs for the OS specific dialogs you may come across while using Eclipse such as browse dialogs. Check that the list of system languages contains your desired language, or else you may see question marks or boxes in the application when the character isn't recognized by the system. You can install missing fonts from any language onto an English OS.

Linux: Set the LANG variable on linux to one of the following OS supported locales plus ".UTF-8". This sets all the localization packs to the desired language. Enter "locale" from the cmd line to see that all LC_* except LC_ALL are set. You don't need to set LC_ALL, consider it an emergency override. You can tweak settings applicable to eclipse: LC_CTYPE, LC_NUMERIC, LC_TIME, LC_COLLATE, LC_MESSAGES. For instance, if you want keep the default binary sorting, where all caps come before lower cases, enter "setenv LC_COLLATE C" or "setenv LC_COLLATE Posix".

setenv LANG <locale>.UTF-8

I18N Phase 3 - Enablement Results

The Enablement quality of CDT is 98.7%.

Enablement tests verify basic support of many types of character, not limited to the 9 languages orf translation. Basic support consists of input, display and processing of NL characters, which includes character joining, and bidirectional algorithms in preparation to display right-to-left text, possibly embedded with English or French left-to-right text and numbers. The CDT portion consisted of 33 tests divided into 9 areas: Standard Build, Managed Build, Scanner Configuration, Class Browser, Make Targets, File management, Debug, Content Assist, and Search. 

The seven countries participating in enablement testing were: China-GB18030, Japan-DBCS, Taiwan-DBCS, Israel-BidiHebrew, BidiArabic, Thailand-Thai, India-Indic. They produced 41 defects of which 5 remain open. The following table shows the updated version of the table in the G11N Interim Report. Thai and Indic results have been added.

Each of the 7 countries ran 33 CDT tests, for a total of 231 tests. Assuming a defect was opened on a failed test case, the pass rate is 228/231, or 98.7%

Suggestions for Future Globalization

Comments came back from the translators, coordinators, and defects with some excellent suggestions for improvements. These can be incorporated into subsequent release design documents.

Externalizing Strings

Translators benefit greatly from well chosen keys, as they can give hints as to the context of the associated label. A key uniquely identified by a number is not useful.

The single quote problem arises when a string is handled by MessageFormat. The single quote should be escaped by another single quote. If there are no arguments, the string is not handled by MessageFormat and the single quote shouldn't be escaped. Eg:

DON’T:  “Message is formatted by ‘{1}’.”
             “Can’t create a file {1}.”
DO        “Message is formatted by ‘’{1}’’.”
             “Can’’t create a file {1}.”

GUI Layout

Consider that the length of labels may expand and decrease during translation. Our mock translation tool only accounted for expansion of text by an equal amount for all labels. A widget in a form layout that attaches to a label in the previous column doesn't have to attach to the label in the same row, the widget with the longest text in that column can be the attach point. In the example layout below, button2's west anchor point should be the widget containing the maximum length of label1 and label2. The mock translation tool used in Internationalization Phase 2 as described in the CDT 2.0 Globalization Interim Report can be revised to grow the strings by a random length to increase chances of catching these types of errors before TVT. 

Cultural Expectations

More emphasis needs to be put on the cultural formatting stage: sorting, parsing, date & number formats. Sorting, dates, & numbers are easy to provide requirements for, but to what extent should we start to support parsing, and at what collation strengths? ie: we should be able to retrieve content assists on inexact matches, since we may not know all the accents used in the code, ie find all the variants of "e" (èéêë) by just using "e" in an English locale.

GB18030

Certain characters in GB18030 are new to Unicode. In Eclipse 3.0.1 and CDT 2.0.2 these surrogate characters were not supported since JRE doesn't support them. ICU4J is an open source api which has the latest unicode algorithms. If requirements for GB18030 become stricter, we may need to look into using ICU4J.

BIDI

Previously SWT didn't support building mirrored applications. Now it does. If this newer SWT is in future releases of Eclipse, then we should be careful when we are using good programming techniques such as SWT.BEGINNING instead of SWT.LEFT in our layouts. Eclipse should not be mirrored unless the text is translated in a BiDi language. However, we should be able to set the locale to a BiDi language and enter text right to left, in an unmirrored Eclipse.

Accessibility

Testing results are not included in this report, but some defects have arisen in some areas and should be considered in future design. Some defects have arisen in particular areas, but should be investigated throughout the application. These areas are:

  • different perspectives have inconsistent shortcuts on the same menu items; 
  • view toolbars don't properly reflect view menus; 
  • mnemonics should be considered as helpful dialog navigators for frequently accessed items; 
  • there are multitudes of composites within composites which prevent screen readers from reading the group label before the widget text.
