CDT DOM Requirements

Features

  • Provide syntactic and semantic information about the code in a project
  • Outline view
    • Show major structural features in the current file (working copy)
    • Used for navigation into the file
    • Also used to give overview of the contents of the file
  • Search
    • Provide the ability to find references to declarations/definitions
    • Provide table of contents of declarations/definitions for the project
    • Provide declaration information about the code in the current selection
  • Content Assist
    • Provide possible completions given the context of the current cursor position
  • Refactoring
    • Allow for structural changes to the code while facilitating checking to ensure semantics do not change

Environment

  • Build Environment
    • Allow features to work independent (as much as possible) of build configuration
      • users should not have to change configs to make elements of the code visible in the model
    • Allow features to work when the source files and the build are at a remote location
      • where no remote file system (e.g. NFS) or cross compilation is available
  • Scalability
    • Allow features to work on large projects without disturbing work flow
      • regular edit/build/debug workflows should work the same independent of project size
      • performance should not degrade to the point where the features do not work "reasonably" on large projects
    • A typical large project would be 100,000 files with an average of 500 characters per file stored in a slow file system (e.g. NFS)
      • Idealy should handle larger, but this represents real projects we've seen
  • Languages
    • Support multiple C and C++ language variants
    • Allow the adoption of this framework by other compiled language IDEs
      • e.g., Fortran, Ada, Cobol
