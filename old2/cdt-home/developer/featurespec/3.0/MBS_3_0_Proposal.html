Introduction

This document presents a proposal for additional functionality to be implemented in the Managed Build System (MBS) for the CDT 3.0 release.  It is divided into 4 sections:

  1. Requirements List: presents a prioritized requirements list of requested MBS functionality.
  2. New Object Model Elements and Attributes: presents the design for the requirements that are proposed to be implemented in CDT 3.0.
  3. New MBS Extension Callback Interfaces: defines new interfaces that can be implemented by a tool integrator in order to provide dynamic information to MBS.
  4. New MBS Extension Points: defines new MBS extension points.

A detailed description of the MBS 2.1 functionality can be found at http://dev.eclipse.org/viewcvs/index.cgi/~checkout~/cdt-home/user/Reference%20Documents/Managed_Build_21/Managed_Build_Extensibility.html?cvsroot=Tools_Project.

There are 3 primary goals for enhancing the Managed Build system functionality:

  1. Multi-platform and multi-version support:  A tool integrator should be able to define a single project-type that contains configurations that can be used with multiple host and target platform combinations, and/or with multiple tool versions.  The user can create one of these project-types using an initial tool-chain and then easily use the project on a different host and target combination, or with a different tool version, by creating new configurations derived from the original configurations.
  2. Tool-chain extensibility and customization:  The user should be able to easily customize the tool-chain used by a configuration by:
    1. replacing the entire tool-chain
    2. replacing individual tools
    3. adding custom steps
    4. specifying different settings, or a different tool, for individual files

In addition, a more sophisticated level of customization will continue to be available by extending the MBS with additional tool-chains and tools using manifest files (plugin.xml).

  1. Usability:  The MBS functionality should be easy to use by the CDT user who needs to define and build his projects.  Extending the MBS should be easy for the tool integrator who needs to integrate her tool-chain (or individual tool) into the MBS.

The goals are intended to take advantage of Eclipse’s strength as a multi-platform, extensible, state-of-the-art, software development platform. 

Requirements List

This list is in priority order (top == highest priority).  The green line represents the cut-off line for CDT 3.0 functionality.

Multi-Version Tool-chain Support:

A tool integrator can support multiple versions of a tool-chain, tool or builder.  The tool integrator can support multiple versions simultaneously, or sequentially.

Simultaneously means that users can choose between the versions when creating projects and adding configurations. The tool integrator may also provide a converter extension (see below) that allows the user to upgrade his project configurations from an older to a newer version of the tool-chain, when the user decides it is appropriate.  The user can also choose to maintain a single project with multiple configurations that use different versions of the tool-chain.  This allows the user to build a project using different versions of the tool-chain.

Sequentially means that the tool integrator requires the user to upgrade his projects to the newest version.

Design notes: A tool integrator defines a version number that MBS uses to match tool references to the appropriate tool definitions.  A tool integrator can define multiple tool definitions that use the same base id and are distinguished by the version number.  MBS attempts to resolve references in the following manner:

  1. MBS looks up the id with the version specified with the reference. 
  2. If not found, MBS looks up all occurrences of the id, and selects a match in the following order:
    1. the id with the closest higher version number that lists the referenced version in its versionsSupported attribute
    2. the id with no version information specified

If a match is found, there are 2 scenarios:

  1. The version is still actively supported by the tool integrator, and the MBS information is loaded.
  2. The version specifies that a conversion to a later version of the tool-chain/tool/builder is required.  At the discretion of the tool-chain/tool/builder, the conversion may occur automatically, or the converter may prompt the user to confirm the upgrade.

If no match is found, MBS displays an error message, and does not load the MBS information.  None of the MBS operations are available on the project (for example, editing build properties, cleaning, building, …).

A version number is specified using the Eclipse PluginVersionIdentifier format, major.minor.service.  The version number gets appended to the element id.  An example of a tool-chain id not using versioning is "cdt.managedbuild.tool.gnu.c.linker".  An example of the same tool-chain id using versioning is "cdt.managedbuild.tool.gnu.c.linker.v.5.1.0".  MBS identifies a version number by the presence of 3 integers (separated by “.”), preceded by “v”, at the end of an id.  A tool integrator must not end an id in this manner for any other purpose.

Tool definitions earlier than CDT 3.0 are considered to have the following version numbers:

  • v.0.0.1 - Tool definitions from a pre-CDT 2.0 project file.
  • v.0.0.2 - Tool definitions from a CDT 2.0 project file.
  • v.0.0.3 - Tool definitions from a CDT 2.1 project file.

Tool integrators should begin their numbering with v.0.1.0 or greater.

When a tool integrator simultaneously supports multiple versions, they must ship multiple tool-chain definitions with the same base id, but different version numbers.

Simultaneously supporting multiple versions is complicated by the fact that a user may or may not have the corresponding multiple versions of the actual tool-chains installed.  The tool integrator should implement the isSupported attribute (see below) so that, by default, MBS will filter out the tool-chains/tools/builders that are not installed.

When a tool integrator decides to no longer support a version of a tool, they continue to ship the old tool definitions and specify the id with version number of the tool definition to convert to in the convertToId attribute.  If changes to the project information need to be performed by the conversion, the tool must provide a conversion extension to perform the conversion.  If no conversion extension is provided, then MBS performs the conversion automatically by simply changing the superClass attribute of the tool definitions in the project to reference the new version.

Converter Extension Point:

MBS allows the tool integrator to register project, tool-chain, tool and builder converters.  These converters can be used for:

  • multi-version support, as described above
  • converting from one project-type to another
  • converting a project/configuration from using one tool-chain/tool/builder to another

See the org.eclipse.cdt.managedbuilder.core.converter extension point definition below for additional information.

Environment Variables and Build Paths:

Tool-chain integrators and users need to be able to define the value of environment variables while the tool-chain is executing.  For the tool-chain integrator, a tool-chain can provide a method to be called that identifies the default environment variables for the tool-chain.  These would typically include the build path variables (“bin”, “include”, “lib”).  These environment variables are defined by MBS for the process in which the builder (e.g., make) runs.

The CDT user can modify the tool-chain provided environment variables in the project properties page.  A new Tab (“Environment”) in the C/C++ Build page allows the CDT user to add additional environment variables, and add additional paths to a variable that represents a path (e.g., “bin”, “include”, “lib”).

Build Macros:

The CDT user can use Build Macros in all configuration properties that accept text.  Build Macros help make a project file more easily shared with other CDT users and more easily maintained.  MBS pre-defines many useful macros and allows tool integrators and the CDT user to define additional macros.  Additional macros can be defined in the MBS preferences page and for individual configurations.  Build Macros are referenced in strings by enclosing them in braces, preceded by a dollar sign.

Design notes: Each build macro should have a “type” that indicates the preferred UI for editing the value of the macro (e.g. a file or path vs. a purely text macro).  File and path macros should use “Path Variables”.  CDT 3.0 will have a path variable manager?  String macros should use Eclipse “Variables”.

When the build-file generator creates the build file (e.g. makefile), should the build macros be expanded, or should the macros be placed in the generated build file and defined in the build process environment?  Should the user be able to select between these alternatives?

Custom Build Steps:

The Managed Build system provides a Custom Build Step tool which can be inserted by the user at any place(s) in the tool-chain of a configuration.  A Custom Build Step can also be used with an individual file, either replacing the tool specified to build the file, or running before or after the tool specified to build the file.  The CDT user specifies the command line, the names of the input files, and the names of the output files.

Custom Tool to File Extension Mapping:

The Managed Build system provides a default mapping of file extensions/file types to tools.  The CDT user can modify the default “file extension/file type to tool” mapping in the MBS preferences and for individual configurations.

Design notes:  This implies that tools need to be assigned into generic categories (e.g. C compiler, assembler, linker, librarian).

Tool-chain Installation Support:

  1. A project-type or tool-chain can provide a method to be called to determine if support for the project-type or tool-chain is currently installed on the system (isSupported attribute).  MBS uses this information in order to filter the choices presented to the CDT user and to inform the user when support needed by their project is not installed.
  2. A tool-chain can provide a method to be called to initialize the tool-chain.  The method is passed the IToolChain interface of the tool-chain, and other interfaces that allow the tool-chain to modify the MBS environment:
    1. A tool-chain can add additional build macros and can set the default value of build macros for configurations using the tool-chain.  This is done through the IManagedBuildMacroManager interface (TBD).
    2. A tool-chain can modify the file extension/file type to tool mapping to be used with the tool-chain.  This is done through the IManagedBuildFileMap interface (TBD).

Remote Building:

MBS provides the ability to use Eclipse on a host system and have the build, in addition to debugging, occur on a remote system.  This would likely use the capabilities to be implemented by the “Remote System (Target) Definition” (Bug #65471).

Option Enhancements:

  1. An option can provide methods to be called to determine if the option is currently visible, enabled and to be used in command-line generation.  By default, an option is visible, enabled, and used in command-line generation.  See the IManagedIsApplicable interface described below (TBD).
  2. A string option can specify the browseType attribute.  This allows the property page to display a Browse… button that allows the CDT user to select a single file or directory as the value of the option.

Tool Input Ordering:

The CDT user can specify the order of input resources passed to tools that process multiple resources (e.g. a linker) from the configuration property pages.

Tool-chain Output Chaining:

A tool-chain specifies how the outputs of one tool in the tool-chain are used as input to another tool in the tool-chain (e.g., are the outputs processed per resource as with a compiler following a preprocessor, or are the outputs processed collectively as with a linker following compiler invocations).

POSIX Make File Generation:

CDT will provide a reference, POSIX makefile generator, in addition to the GNU makefile generator that is provided in CDT 2.1.

It is not known at this time whether all of the proposed MBS functionality can be implemented using a POSIX makefile.  It is possible that the POSIX makefile generator, or future MBS builders, may not be able to implement all of the semantics defined by the build model.  When this is true, the build/make file generator must indicate in the build output window and the build log file which part(s) of the project build model could not be honored.  It would also be desirable to allow builders to somehow specify the functionality that they do not support so that the MBS could disable the UI for those features (TBD).

Extract GNU Tool-chain Definition into its Own Plug-in:

Remove the GNU tool-chain definition from the base MBS code.  This allows CDT providers to decide whether or not to include the GNU definitions in their distribution.

Multi-Instance Tool Support:

A tool integrator can also support multiple “instances” of the same version of a tool-chain.  Compared to multi-version tool support, this is a “lighter weight” approach, uses a different UI (TBD), and is typically used when the tool-chain version instances support a semantically equivalent set of tools and options, but may differ, for example, in the target platform and where the tool-chain is installed on the system. 

If the user has multiple instances of a tool-chain installed, the user can select the instance that they want to use per configuration.

Design notes: To support multiple instances of a tool version, the tool integrator uses the toolChainInstanceSupplier attribute.  This attribute specifies an extension callback that implements the IManagedToolChainInstanceSupplier interface as described below (TBD).

Additional Tool-chain Installation Support:

The tool integrator can add additional default configurations to a project-type from a separate manifest file.  This allows a family of tools to add configurations as tool-chains are installed.  It also allows tool providers (e.g. profilers, code coverage tools) to add configurations to a project-type that enable the use of their tool.

Build Log Generation:

The CDT user can request that a build log be written to the output directory.  The default is set from the Managed Build preferences page setting when the configuration is created and can be modified from the configuration property pages.

Additional Option Enhancements:

  1. An option can provide a method to be called to return its default value.  This supports the case where the default value of the option changes based upon the value of another option(s) or some other environment setting.
  2. The value of a particular instance of an option is dependent upon its current context.  The Managed Build system supports option value “inheritance”.  The inheritance hierarchy is “option type default” -> “option default” -> “configuration setting” -> “resource specific setting”.  For option types other than stringlist, a value specified at a lower level in the hierarchy overrides a value specified at a higher level.  If a value is omitted in a lower level, the value is inherited from the closest ancestor with a specified value.  Stringlist options support the special build macros $(inherit) and $(noinherit).  If neither $(noinherit) or $(inherit) is specified as an entry in a stringlist, the stringlist entries of the next higher level in the hierarchy are automatically applied to the end of the list.  If $(noinherit) is a member of the list, the stringlist entries are not inherited from the next higher level.  If $(inherit) is specified in the list, the stringlist entries of the next higher level in the hierarchy are applied at that point in the list.  $(noinherit) takes precedence over $(inherit).  Design notes:  Stringlist entries do not inherit by default in CDT 2.1.  When the default is changed to inherit, a project conversion step will need to add $(noinherit) to all existing stringlist values.
  3. An option can provide a method to be called to validate a new value for the option.  By default, the Managed Build system accepts any value appropriate to the option type.
  4. An option can register observers/listeners that want to be notified when the value of the option changes.  The client must implement the TBD interface described below.

Property Page Enhancements:

  1. A property page provides a visual cue (bolding) when a property value is different from the current default value of the property
  2. A property page provides an easy mechanism for setting an option back to its default value in the current context.

Context Sensitive Help:

  1. An option can specify a help “tip” to aid the CDT user in setting the value of the option.
  2. A project-type can provide a method to be called to display help from the New Project dialogs.
  3. A tool-chain, tool, option category, and option can provide a method to be called to display detailed help information from the configuration property pages.

Project Wizard Enhancements:

A project-type can optionally add tab pages to the “Additional Project Settings” page to allow the user to select various options, and can optionally populate the new project with code that is specific to the type of project.

Additional Dependent Project Support:

The outputs of dependent projects are automatically used by the project in a manner appropriate to the project-types.  For example, if the dependent project is a static or dynamic library, the output of the dependent project is added to the linker/archiver/librarian tool.  If the dependent project is a dynamic library, the library path is added to the debug configuration.

Additional Configuration Support:

  1. The CDT user can change the tool-chain to be used by a configuration.  All appropriate settings are copied from the current tool-chain to the new tool-chain using the concept of “compatible” tools:

·        Copy appropriate tool-chain attributes.

·        If the new tool-chain contains the same tool as the old tool-chain, copy all of the tool options.

·        If the new tool-chain contains a tool that is compatible (i.e., defines a common set of options) with the old tool chain, copy all of the compatible tool options.

·        Copy all of the Custom Build steps from the old tool-chain to the new tool-chain.  Make “best effort” at placing the Custom Build steps in equivalent positions in the new tool-chain.

Design notes: compatible options use commonly defined alias-ids in addition to the unique id defined for the option.

  1. The CDT user can add additional configurations based upon an existing configuration, using a different tool-chain for the new configuration.
  2. The CDT user can export a make/build file based upon the current state of the configuration at any time via an explicit “Export Build File” menu item.
  3. The CDT user can override or add to the default “Clean” script for a configuration.
  4. The CDT user can select a set of configurations to modify by selecting a “category” of configurations.  Category names are defined by the tool integrator and can be added to by the CDT user.  When the selection contains the same or compatible tools, the user can view/modify the options that apply to all of the configurations in the selection.
  5. The CDT user can select a configuration and replace all of the appropriate build settings with the settings from another configuration.

Additional Build System Support:

  1. The Managed Build system defines a default “build intermediate output” directory that is a sub-folder of the project folder using the name of the configuration.  Note that the default for the “build output” and “build intermediate output” directories are the same.
  2. The CDT user can change the “build output” directory.
  3. The CDT user can change the “build intermediate output” directory.
  4. The CDT user can insert an additional tool(s) at any place(s) in the tool-chain.  The tool can come from another tool-chain or a standalone tool definition defined in an installed managed build manifest file.
  5. The CDT user can rebuild an individual resource without performing a project build.  For example, the user can compile an individual source file to check for compilation errors.
  6. The CDT user can perform a build up to a user specified build step – for example, compiling all files but not linking.
  7. The CDT user can select a set of configurations to build.  Design notes:  Eclipse 3.0 added the concept of “scoped builds” – is that what we are looking for?
  8. A tool-chain can specify the build order of the tools in the chain, or it can allow the Managed Build system to determine an order based upon tool inputs and outputs.
  9. A tool-chain can invoke a tool multiple times even when the tool is not associated with a type of project resource that can appear multiple times in the project (e.g., the linker).
  10. A tool-chain specifies whether the failure of the tool causes the build process to continue or stop.
  11. A tool can provide a method to be called that enumerates the input file dependencies of a particular project resource.  The Managed Build system provides default implementations for certain categories of tools (e.g. C/C++ compiler).
  12. A tool can provide a method to be called that enumerates the outputs of a particular project resource.
  13. A tool can specify a message to be displayed in the build log immediately before the invocation of the tool (e.g., “Compiling with Intel C++ v8.0…”).  By default, MBS will display the message “Invoking tool-name (tool-id)…”    Design notes:  Does this make the build log too wordy?  If we try to use this to identify output for error parsers, are there internationalization issues?
  14. A tool can produce multiple kinds of output (e.g. .so & .a)?  Do we need the concept of a primary output and secondary outputs?
  15. A tool/builder specific error parser is added to the end of the error parser list, if not already present, when a project resource is defined to use the tool/builder.  The error parser can be removed by the CDT user, and is automatically removed when there are no more project resources that use the tool/builder.
  16. An option can specify a precedence order value that determines where it is placed on the command line relative to other options.  Options with the lower precedence values appear before options with a higher precedence value in the generated options.  The default precedence order value is 0.  Options with negative precedence order appear before options with no precedence order.  Options with precedence order values > 0 appear after options with no precedence order.

Additional User Preferences:

  1. The CDT user may remove installed and possibly-installed (i.e. the project-type does NOT implement the method of verifying that its support has been installed) project-types from future consideration by turning them off in the MBS Preferences page.
  2. The CDT user may remove installed and possibly-installed (i.e. the tool-chain does NOT implement the method of verifying that its support has been installed) tool-chains from future consideration by turning them off in the MBS Preferences page.

Additional Property Page Enhancements:

  1. Options support a guarded-string option type.  Guarded-string options are used for a tool command that turns a tool feature on or off, and when “on”, optionally takes a user-defined value.  An example is when a tool can optionally generate a particular output file and the user can optionally specify the path of the file. The UI representation for a guarded-string option is a check box followed by an edit box.  The edit box is only enabled for input when the check box is “checked”. 
  2. Options support a Table (2 column) option type for options that take 2 string values.  The user can edit the cell contents, and add and delete rows.
  3. The tool integrator can override the standard property page display and provide a custom page.
  4. An option can provide a custom editor to be used to view or change the value of the option.
  5. An option can request that it use a Managed Build system built-in custom editor.  Examples of such editors include a string list editor and a file/path browsing editor.

Miscellaneous:

  1. A project (and its child elements) can store private information in the Managed Make project file (.cdtbuild).
  2. The CDT user need not specifically choose between a C and C++ project.  CDT should be able to assign intelligent per-file defaults based upon the file extension, and allow the CDT user to specify C vs. C++ if necessary.  Design notes: The main issue here is how to handle file types where it is not obvious whether they are C or C++ files – e.g. .h files & .obj files.  Also, to do this impacts more than just the MBS.
  3. A tool-chain can provide property setting “wizards” that modify sets of tool options in order to reach an overall user-desired goal.  An example would be a “Most Highly Optimized Build” wizard that set options on multiple tools.  Design notes: Where do these buttons appear in the UI?  How are they defined?
  4. Share/Enforce Compiler Settings Across a Team – Part of the Eclipse 3.1 proposed functionality is “For team-wide uniformity, some preferences settings (e.g., compiler options) need to be shared (or imposed) across teams”.
  5. MBS allows certain attributes to be overridden that would have a large impact on the scanning of tool output – for example, the tool command and the make command.  Does MBS allow the user to override everything that they need to in order to make this work?

New MBS Extension Points

org.eclipse.cdt.managedbuilder.core.converter

This extension point allows a tool integrator to contribute a project-type/tool-chain/tool/builder converter to MBS.  Converters can be used to upgrade to newer versions, change the project-type, or to perform conversions between different tool chains.

A Converter specifies in its extension definition:

  • The class that implements the IConvertManagedBuildDefinitions interface
  • The MBS version the converter was written for
  • The “from” tool-chain/tool id(s) – a regular expression
  • The “to” tool-chain/tool id(s) – a regular expression
  • A priority value in case MBS finds more than one match for a conversion

A converter cannot be invoked unless both the “from” and “to” tool-chain/tool definitions are installed into CDT.  Old tool-chain/tool/builder definitions, that are to be used in conversions only, are marked by the tool integrator with the convertToId attribute set to the id of the new tool-chain/tool/builder definition.

MBS invokes a converter:

  • During MBS version upgrading when the tool-chain/tool is found to require conversion.
  • When MBS loads a project file and determines that a referenced tool-chain/tool has the convertToId attribute set.  At the discretion of the converter, the user may be asked to confirm the conversion.  If the user does not confirm, the project is loaded in a read-only mode, and any changes made to the MBS information will be discarded when Eclipse exits.
  • If conversion is invoked explicitly from the UI.  When conversions are possible for a selected project, the project context menu should display a “Convert To  >” menu item with a submenu listing the possible conversion targets.

Converters use the MBS model public interfaces (IManagedProject, IConfiguration, …) to modify the project definition “in-place”, and so could be affected by MBS model changes.

This is the definition of the IConvertManagedBuildDefinitions interface.

 

public interface IConvertManagedBuildDefintions {

   public toolChainPreConvert(IToolChain toolChain, String fromId,

String toId, boolean userHasConfirmed);

   public toolConvert(ITool, String fromId, String toId,

boolean userHasConfirmed);

   public tooChainPostConvert(IToolChain, String fromId, String toId,

boolean userHasConfirmed);

}

 

The conversion process performs the following steps per tool-chain:

  1. Calls toolChainPreConvert if the tool-chain versions are different
  2. Calls toolConvert for all tools, and the builder, in the tool-chain where the versions are different
  3. Calls toolChainPostConvert if the tool-chain versions are different

 

When a tool integrator conversion is invoked while an MBS conversion is also taking place, MBS will invoke the converter with the appropriate object model based upon the MBS version that the converter was written for.  If the converter was written for the old MBS version, the converter is called with the old version of the MBS object model.  If the converter was written for the new MBS version, the converter is called with the new MBS model.
