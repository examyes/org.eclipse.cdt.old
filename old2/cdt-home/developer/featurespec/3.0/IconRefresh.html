1. Introduction

The IBM media design team has submitted a proposal to update the CDT icons to be more consistent with Eclipse 3.0/3.1.

This doc presents the changes and options and is a placeholder for feedback.

Specifically I had some concerns about the following icons and the changes I've made: library icons [3/58] tool configuration icons [44-48/97-101]

Responses

Dave:

Q: 27, 29, 30, 32-35, 37, 39, 46, 59, 65, I'm not sure how these icons are used and what they mean or represent?

A:  These are all used by the CDT debugger to represent stuff extracted from the executables

Bogdan:

95 - namespace, 107 - struct, 110 - typedef, 112- union, These are all used in the same context (CView, OpenType dialog) so it would make sense to adopt a similar style.  I think that the originals are more appropriate.

85 - header file, 86 - header folder, 100 - assembler file, Used in the same context as source files (57), should adopt similar convention.

111 - IProblem Marker. Coloured orange on purpose - in order to match up to orange annotation.

Thomas:

Some general comments from my look through the listings that I'll throw out for comment ... if I didn't comment then I liked the new icon proposed:

#26 - signal_obj -  I personally don't feel that the new icon represents a "signal" concept which is an asynchronous communication into the program (hence the "lightning"). I'm not tied to the lightning, but can we ask them to try again?

#28 - var_env -  I'm not a fan of the old icon, but at least it gave you an indication of that it was an "environment" variable, the new one is just another dot.

#31 - var_pointer - #38 - vard_pointer -  The new icon doesn't "feel" in line with the other variable icons. I think that they were basing this on the same image/concept as the instr_ptr (current instruction pointer used in the editor tray).

#49 - target_obj -  I really like the new icon(s).

#56 - build menu -  This is somewhat tied to the #49 and I think that iconographically it would be nice to have a tie between them.  Right now they are very distinct

#85 - hfile_obj - #86 - hfolder_obj -  I don't think that the new icons are representative at all.  I like the concept of marking header files/folders the same as C/C++ files and folders. This graphical representation doesn't mean anything (to me anyway).

#110 - typedef_obj - #112 - union_obj -  I think that if enumerations are going to be marked with an E (which I like) then typedefs/unions should be marked similarly ... or at least similarly to structs. The suggested icons don't map consistantly and since these items are "similar" then the iconography should also be similar (arguable point I'm sure).

Mikhail:

#1, #2 Address breakpoint - I don't like the current one with "A", but I'm not sure that the new image can be associated with an address breakpoint.

#6 org.eclipse.cdt.debug.ui/icons/full/obj16/disassembly_obj is no longer in use.

#7 org.eclipse.cdt.debug.ui/icons/full/obj16/ is no longer in use.

#15, #16 images for shared libraries with or without debug information. The new image is used for regular libraries. To indicate that library or executable contains debug information it would be nice to have an overlay image - the green bug, for instance, instead of "S".

#26 signal. I would prefer the old image.
